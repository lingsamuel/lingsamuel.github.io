我的特别版 MOD 体系

Ling Samuel • 2020-10-12 • game  • skyrim  modding 

2020 年国庆节我都干了什么——一个大型上古卷轴 5 MOD 指南。

因为某种神秘的契机（其实我忘了为啥），以及在博德 3 里遇到了一些 bug，国庆节我重新捣鼓了一下上古卷轴 5 的 MOD，当然是特别版的。

传奇版的 MOD 我前几年捣鼓了很久，但 CTD 的概率还是很高，我觉得还是老引擎的问题。特别版在装了更多 MOD 之后 CTD 竟然还是基本不存在，感觉确实是传奇版的引擎问题了。

本指南的目标是：一个真实的、RPG 感浓郁的、有沉浸感的世界，与少女卷轴的关系局限于使用一些美化 NPC 与捏人的 MOD，任何奇异的服装 MOD 乃至更甚者与本指南绝缘。

警告：按照本指南进行 MOD 组装，可能花费你非常多的时间，做好消耗半个月甚至一个月的准备！

MOD 寻找

不得不吐槽，Nexusmods 是真的很难用。我试过开两个网页，然后用同样的规则翻到同一页，内容居然不一样……

总之，我寻找 mod 的主要方法是扫榜单，扫了四十来页吧，100K+ 下载量的都被我扫完了。另外，我还看了一下 STEP Project。

MOD 分类

MOD 大致上分为如下几类：

  • 纯粹的模型与贴图优化。

    这一类 MOD 基本上不新增脚本，有的也不会新增 ESP。例如各种高清材质包，SMIM，等等。这种 MOD 即使冲突了也一般不会导致 CTD，最多是贴图丢失之类的。

    这种 MOD 安装起来最省心，一旦出问题了排查起来也基本上是能稳定复现的，不像脚本出问题了可能是随机现象，排查起来痛苦得要命。

    如果所有 MOD 都如画质 MOD 一样安装方便效果斐然，那世界该多美好啊。

  • 世界修改。

    例如著名的 JK’s Skyrim，以及 Realistic Water，各种天气光照修改，3DNPC，等。这些内容修改与特定的 MOD 可能会产生冲突。

  • 独立随从。

    我只使用了自制的独立随从。独立随从一般不会引起冲突，但可能会导致一些 MOD 的修改，如武器装备、属性等不能覆盖到独立随从，或者因为 Perk 或法术等被修改了导致实际生效的效果并不完全按照作者所想。一般来讲独立随从会有自己的设置，覆盖不到也没关系，问题不大。

  • 人物美化，动画，以及捏人相关。

    此类 MOD 一般都含有很多 ESP，多个美化 MOD 使用的话，除非真的很作死，一般不会引起 CTD。但会有其他问题，例如头身肤色不一致，黑脸等。多美化 MOD 公用，冲突率极高。人物美化可以说是万恶之源之一了，后续会有专门的 Patch 步骤。

  • 新增武器装备法术。

    • 纯粹新增，而不更改现有 NPC。

      一般是通过 Leveled List 或只允许打造（Craft only）实现，可以使用 Merged Patch 和 Bashed Patch 解决。

      对于武器和装备来说，只要使用的还是原版的属性体系，一般是不会冲突或产生问题的。所以我特别不喜欢大规模修改数值的 MOD。

      对于法术而言，比较常见的可能是光照和效果上会有一些小问题。例如 AFT 的法术对友军无伤 MOD 是针对特定法术一个个修改的，显然并不能自动套用到新增的法术上。

      *二者可能兼具的问题：一些 Perk 或者效果不能套用到它们上。*这个没什么通用的解决办法。

    • 与 NPC 武器装备有关的。

      除了类似 Immersive Armor 这种用脚本实现的，例如 Immersive Weapons 等，大概率与人物美化冲突，依赖各种 Patch。

      好消息是这部分的修改是很稳定的，不像上面那种，某种 Perk 对某个装备不生效，一般还发现不了……而脚本实现的，出了问题都不知道咋回事。

  • 各种影响 NPC 的补丁。

    同上，例如 Immersive Weapon 和美化包，就需要 Patch。

    还有的例如新增大量可结婚 NPC 的 MOD，是通过 ESP + 脚本实现的，也需要补丁。

  • 各种大修。

    只要不是修改了同一方面，一般不会冲突。

    当然如 CACO 混合 Ordinator 这种，配合补丁是完全可以共存的，符合下文提到的原则。目前，CACO，CCOR，CCO，Ordinator，AGO，我都一起用了，暂时没发现有问题。

  • 新增玩法。

    例如 Frostfall、iNeed、Hunterborn。这种一般都含有针对游戏内某一方面的脚本功能，尽量一个方面只选择一种，避免脚本冲突或执行某些功能时脚本压力过大。

    需要注意玩法 MOD 与大修 MOD 的兼容性。如生物 MOD 和 Hunterborn。

  • Player’s Home。

    除非修改了同一地点，一般不会冲突。但有补丁可能会更好。例如 CACO 给很多房子都提供了补丁。

  • 新增任务链、地图等。

    基于原版世界单纯地新增任务，一般不会有特别影响体验的冲突问题。

    但如 Falskaar、Bruma、Cutting Room Floor 等新增了地图的，均大概率与各种人物美化、天气光影、各种大修、玩法 mod 冲突或不生效。谨慎安装。

  • Encounter Zone & Dungeon & Radiant

    冲突概率很大，遇到了就挺头疼的。

    除了 Redone 和 Creatures 等 MOD 之外，我也是第一次使用 Sands of Time 这样的大型 MOD，没什么经验。

    （2020-11 更新）：Sands of Time 使用体验并不佳，详见后续。

  • 战斗系统。

    说实在的，这个真没救。别尝试了……

一些原则

下列原则并不仅适用于本指南里提到的 MOD。

  • MOD 的覆盖关系：如果两个 MOD 可以并存，那么让专门的 MOD 覆盖大范围更改的 MOD。

    例如，Ordinator 的技能树修改需要被 CACO 与 CCOR 覆盖。

    SMIM 这种覆盖面非常广的材质包，应该被各种专门的贴图与材质包覆盖，例如我专门下载了神龛、神像、奥杜因之墙等高清包。

  • 永远都要阅读作者的兼容性说明，以及 Posts、Bugs。

    但不可认为作者没说的就是兼容的。MOD 作者也不是万能的，他们也不可能知道所有兼容性问题。

    但是，因为我挑选的都是比较热门的 MOD，Posts 和 Bugs 页面就很可能有相关的置顶帖或者冲突报告。

  • 谨慎选择 Script-Heavy MOD。

    有脚本的 MOD 装多了之后，必然会出现各种各样的问题，不要妄图解决——因为最大的可能一般是老滚的脚本系统撑不住了……

    尽量不要安装效果不明的 MOD、查看并搜索每一个 MOD 的脚本情况。

  • 老 MOD 都可能有 Bug，忍受它或者不安装它。

    例如，Frostfall、iNeed、Immersive Creatures、CCO 等都有很多 bug。

  • 永远不要安装修改面巨大的奇怪 MOD，除非你只使用它。

    没错，我说的就是 SexLab 等。我个人对此种 mod 深恶痛绝。想 ghs 请去专门的游戏。

  • 永远不要在存档中途增删 MOD，除非你很确定不会产生问题，并且知道如何清理存档。

  • 开启 SKSE 损坏脚本清理的功能。

  • 开启 SKSE 新存档而不是覆盖存档的功能。

  • 除非你使用整合包之后不进行任何改动，否则，永远不要试图使用整合包。

    不然你必定会遇到各种各样奇怪的问题，然后花费茫茫多的时间去调试。

  • 提前引入汉化 ESP。

    因为文本可能被 Wrye Bash 或者 xEdit 读取到 Patch 中。若不提前引入汉化 ESP，会导致后续对 Patch 的汉化工作很难进行。

  • 补丁包尽可能紧贴原包。例如 WICO 的各种补丁，尽量紧随 WICO 之后。

    如果排序类似于 WICO - Shiva - WICO Patch 这样的顺序，后续的补丁制作会想吐血。

MOD 选择与取舍

MOD 选择也是有一些基本原则的。

  • 尽量避免有容易触发的已知 bug 的 MOD。

    游玩时小心翼翼地在心里重复这些触发条件真的很令人烦躁。

    但如果只会遇到一次，那就可以接受。如我选择的风宅 MOD 唯一要求是不要购买额外的配件，以及 navmesh 有冲突问题。

  • 如果是 Script-Heavy 的 MOD，尽量做出取舍。

    例如，我没有使用 VioLens，但我使用了 Wet & Cold。

  • 能不装的 MOD 就尽量不装。

    实际上，很多 MOD 都只是看着美，玩起来的时候实际接触并不很多甚至讨人厌，这种 MOD 都可以不装。

    例如：Animated Eating Redux，ELFX Hardcore。

几个简单的评分规则：

  • 仅包含 BSA 与用于载入 BSA 的空 ESP，BSA 内仅含 Texture 与 Mesh。

    顶级，让人放心的 MOD。一般不会产生恶性或潜在的难以发现的 Bug。

  • 仅包含松散的 Texture 和 Mesh 文件。

    顶级，让人放心的 MOD。虽然 Loose File 优先级永远高于 BSA，但 BSA 也使得覆盖变的难以发现（主要是 MO2 的 BSA 覆盖功能不能用了）。

  • 无脚本，ESP 非空。

    简单 MOD。此类 MOD 的恶性冲突都是可预见的。

上述的 MOD 一般来说可以想装多少装多少，且不怎么影响游戏的性能。

  • 有脚本，ESP 非空。

    需要警惕。重点关照的 MOD。能少装就少装。

特别的，facegendata 文件夹需要额外的关注。

关于引擎的一些说明

脚本。上古卷轴的脚本系统真的非常烂且性能低下，有脚本常驻的 MOD 能少用就一定要少用。

测试

所有的测试都是不完善的。但可以针对 MOD 的核心功能进行测试，并且不要一次性安装海量的 MOD。

安装过程中不断测试所花费的时间，是为了减少安装后的调试时间，甚至是避免开始游玩后才遇到 Bug 而浪费时间。

此外，测试还强制要求你很了解你安装的 MOD 修改了什么，这使得你更了解你的 MOD 内容，避免莫名其妙的错误。

了解你的改动

手动制作 ESP 的补丁时，有一个灵魂问题要时刻叩问自己：

多个 ESP 存在记录冲突时，其中一个 ESP 对某条记录没有修改，到底是没有修改（unchanged），还是要求它保持原样（require unchanged）？

一般来讲，除了很容易确定的情况，例如 Leveled List、名字的汉化、NPC 的美化，大多数时候这都是一个很困难的问题。谨慎修改。

总的来说，我只推荐合并 Leveled List、NPC 装备技能等容易确定的属性。其之外的记录并不推荐随意合并。

汉化 —— xTranslator

用 Wrye Bash 生成的补丁会导致中文出现乱码，可以使用 xTranslator 导入对照 ESP 来生成补丁。Oblivion 的文档显示 Wrye Bash 有个设置（Settings - Plugin Encoding）可以解决这个问题，但据说没用。我没有尝试。

此外，有的 MOD 如 Wheels of Lull 在 Nexusmods 上没有汉化，但已知有 LE 版的汉化，也可以通过导入对照 ESP 来生成汉化补丁。

有的 MOD 的汉化补丁过期了，也可以通过这个方法解决。

一般只要 MOD 没有大改，文本内容是不会变的。用旧版的汉化 ESP 生成新版的补丁一般不会产生问题。

此外需要注意的是，例如 Ordinator 会有很多补丁。这些补丁一般是没有汉化的，而排序问题会导致汉化失效。

有时候，xEdit 将汉化失效的记录标记为红色，而只能标记为绿色。对于一个新增记录，由于其并不包含在本体中，如果只有两个 esp 修改过它，那么汉化失效并不会被检测为红色冲突。例如 a.esp 新增了一个记录，此 esp 是汉化过的。而 a - patch.esp 是没有汉化的补丁包，也修改了同一记录。因为一共只有两个 esp 修改过这个记录，并不会被判定为红色，而只是绿色。这时，自动生成的 Patch 包就不会处理这种条目。

有的补丁包还会修改文本内容，使其与原文本含义不一致。例如 Elemental Destruction 的 Ordinator 补丁，就给几个 Perk 新增了几种效果。此时如果盲目用 xTranslator 去刷汉化，就会导致文本与实际效果不一致。

有一个特殊情况是，有的 MOD 看似没有 ESP 记录冲突，但实际上冲突依旧存在。常见情况是全人物美化如 WICO 修改了一个 NPC，其他美化虽然没有修改这个 NPC，但资源文件等已经不适用于 WICO 的改动了。这种情况是最难预先发现的，只能遇到一个解决一个。

Manual Patch

自动化的补丁是远远不够的。你需要手动解决一切错误、遗漏的部分。这部分非常消耗时间，但这一步才真正的 make things work properly。

我不推荐直接对 Bashed Patch 或 Merged Patch 进行修改，而推荐新建一个 Patch。这两个 Patch 都是自动生成的，应该保留其能被自动生成的特性。而且这二者的修改面一般都非常广，并不是所有条目都需要被手工解决。

不要迷信 Merged Patch 与 Bashed Patch

Bashed Patch 依赖 MOD 的 Tag 进行生成，而实际上并不是所有 Modder 都了解相关信息的，因此 Bashed Patch 并不是万能灵药。

并不是 Merged Patch 和 Bashed Patch 里没有出现的记录就没有冲突。我永远推荐你检查所有冲突的条目。

绿色标记不等于没有冲突

不要盲目信任 xEdit Show Conflict 的 Filter 颜色。永远都要检查补丁的全部条目。

绿色标记只是代表某个条目在出现之后只被一个 esp 修改过，但并不代表这个修改就兼容其他的修改。

汉化补丁

如果你需要汉化补丁，而汉化过于老旧，不适用于新版，那么你可以使用 xTranslator 将旧版 ESP 内的汉化词条导入到新版中。

Plugins Count Limitation

0xFD - 253 个 MOD 数量限制。

特别版引入了 ESL 机制，这个机制极大地减少了补丁的数量压力，然而 MOD 装多了之后还是有可能产生超限问题。

第一个解决方法：将 ESP 标记为 ESL。

首先提一个 ESL Flagged ESP，大多数 Patch 包都是这种格式。这种 ESP 依旧可以正常排序，但依然占用 FE 的优先级，避免了 ESL 文件排序在 ESP 前面的问题。只需要在 xEdit 的 File Header - Record Flags 里增加 ESL 即可。有一个小脚本可以筛选出所有可以转成 ESL 的插件。

第二个解决方法：Merge Plugin。

实际上这并不是一个足够优秀的解决方案。它具有肉眼可见的缺陷：

  1. 有的 MOD 的 BSA 文件与 ESP 同名，这种情况下会自动载入 BSA。Merge 之后 ESP 文件名变了，BSA 就不载入了，需要手动合并 BSA 或解压成 Loose File。
  2. 有的其他 MOD 依赖特定文件名作为 Master。这是最常见的不能合并的情况。
  3. 合并后的 esp 不具备排序的可能。

但一些类似于 NPC 跑动速度和玩家一致，移除径向模糊等小补丁，其排序绝大多数是无所谓的，因为一般没有 mod 对这些东西进行修改。这类型的插件就可以合并。

特别的补丁 - 自行制作

PetKing 这个 MOD 加入了很多奇奇怪怪的宠物，但他们是可以敌对且不死的……

PetKing 加入的猫猫狗狗可能挡住 Helgon 的路，一般不会产生影响，只是会导致马车碾过去而已。

注意事项、一些问题及其解决方法

存档问题

老滚 5 存档的覆盖机制有 bug，可能产生损坏的存档。尽量存新档而不是覆盖。

黑脸问题

美化 MOD 冲突。请精心修改冲突部分。

NPC 不动、浮空

安装 Increase Actors in a cell。

夜里瀑布太亮

打补丁。

可以尝试一下 这个。

我之前不知道在哪还看到过一些补丁，但找不到了，遗憾。

If you use WATER or Realistic Water Two (either recommended!), install one or the other AFTER SMIM and let it overwrite all SMIM files. If you use SkyFalls and SkyMills (recommended), install it BEFORE SMIM and then let SMIM overwrite all files. Otherwise, one of the windmill fans will remain 2D. If you use Mindflux’s ENB Particle Mesh Fixes, let SMIM overwrite everything because SMIM already includes all his fixes.

地平线处一片黑

常见于暴风雪的天气。据说和 ENB 的 Mist 特效有关。下次遇到研究研究。

开局问题

Helgen 的过场脚本量非常大，即使是没有问题的 MOD 组合，也可能在 Helgen 滑铁卢。Helgen 的海量脚本导致在开新存档的时候，可能由于脚本引擎压力过大，很容易出问题。最常见的问题是发现乌弗瑞克的衣服没有变成 Immersive Armor，哈达瓦还消失了（我还遇到过哈达瓦站在路边、开局屏幕刚亮马车就将我击杀、哈达瓦突然从天而降等各种情形）。哈达瓦如果刚开局不在，后续貌似也会在某个时刻出现。但我推测开局的部分脚本和哈达瓦有关，以至于后面转弯的时候会出现 bug，转弯只转了一半然后卡在石头上……

顺带一提，因为我不太喜欢 Alternative Start，我的解决办法是……不断刷开局……我还有一次遇到不断刷开局然后选择返回主菜单时 CTD，完全没有头绪。

此外，我也不知道是装了哪个 MOD 导致貌似开局的马车多走了很长一段路……

结语

本 MOD 体系的脚本量很大，且并不能称之为 bug-free，游戏启动时日志里会有海量的报错，也不知道是不是没问题。不过实际游玩中的报错貌似不多，我在日志里经常能看到很多连续的存档日志（VM is freezing），证明两个存档之间应该没出过什么问题。总得来说应该还是比较稳定的。

.Net Script Framework 有提供 Crash 日志，但并不是任何时候都有用的。

从天际里想到的

我越来越觉得，一个设计良好的可扩展系统，没有 partial patch 与 patch rule，是不可接受的。

Partial patch 解决了类似于美化 MOD 和武器 MOD、汉化 ESP 冲突的问题。Patch rule 解决了我提到的 unchanged 或 require unchanged 的问题（实际上 unchanged 这一情形也被 cover 在 partial patch 里，但 patch rule 能做更多）。
